## 1. 현재 수준 평가

### 1.1 잘 되어 있는 부분

| 항목 | 상태 | 근거 |
|---|---|---|
| **Tool-using LLM 에이전트** | ✅ | 기능1~4 각각 Claude `tool_use` 루프로 MCP 도구를 동적으로 선택·호출 (`agents/agent1_summary.py` ~ `agent4_comparison.py`) |
| **MCP 프로토콜 준수** | ✅ | 에이전트가 DB/SDK를 직접 호출하지 않고 `mcp_client.call_mcp_tool()`만 사용 |
| **느슨한 결합** | ✅ | 서브 에이전트 간 직접 통신 없음. `shared_store`(SQLite)를 통해 Orchestrator가 중간 결과 전달 |
| **순차·병렬 혼합** | ✅ | 기능1→2→3 순차, 기능4는 기능1 완료 즉시 백그라운드 병렬(`asyncio.create_task`), 기능3 내부 4체커는 `asyncio.gather` 병렬 |
| **Human-in-the-Loop** | ✅ | `human_approval_gate`로 기능2 초안 후 팀장 검토 강제. 최근 추가된 "수정 후 품질 재체크" 기능으로 팀장이 기능3 결과를 보고 수동 반영 가능 |
| **결과 표시 일관성** | ✅ | `format_emp_references_deep`로 emp_id → "이름 직책" 변환, `employees.json`을 신뢰 소스로 비교표 이름 보정 |

### 1.2 한계

| 항목 | 상태 | 설명 |
|---|---|---|
| **Orchestrator는 비-에이전틱** | ⚠️ | `orchestrator.py`는 LLM이 아니라 고정된 Python 흐름(if/await 순서가 코드에 박혀있음). Anthropic의 구분으로는 "Agent"가 아니라 "Workflow(미리 정의된 경로 + LLM 단계 조합)"에 가까움 |
| **자기 평가/재시도 루프 없음** | ⚠️ | 기능3이 critical 경고를 내도 기능2가 자동으로 재작성하지 않음. (방금 추가한 재체크는 "팀장이 수동으로 수정 → 기능3만 재실행"으로, 기능2는 관여하지 않음) |
| **RAG가 사후 검증에만 사용됨** | ⚠️ | `retrieve_grade_criteria`/`retrieve_guideline`은 기능3·4에서만 호출. 기능2(초안 생성)는 가이드라인을 참고하지 않고 작성 후 사후 검수만 받음 |
| **장기 메모리 부재** | ⚠️ | `eval_history.json`이 정적 mock 데이터. 올해 승인된 draft가 내년 평가의 `get_past_feedback` 입력으로 누적되지 않음 → "성장 추적" 기능이 명목상으로만 존재 |
| **데이터 결손에 대한 적응 없음** | ⚠️ | CLAUDE.md 엣지케이스("1on1 이력이 Q1만 있는 직원")는 프롬프트 수준 처리에 의존. Orchestrator/에이전트가 데이터 완결성을 판단해 흐름을 조정하지 않음 |
| **에이전트 간 협상/위임 없음** | ⚠️ | "Multi-Agent"라기보다 "단일 LLM 호출 4개를 파이프라인으로 연결"에 가까움. 한 에이전트가 다른 에이전트에게 작업을 위임하거나 결과를 비평하는 구조 없음 |

### 1.3 종합

> **현재는 "Tool-using LLM 단계들을 결정론적 파이프라인으로 묶은 Workflow" 수준.**
> MCP 연동, 병렬/순차 혼합, Human Gate 등 *구조적* 요건은 설계 문서대로 구현되어 있으나,
> *자율적 판단·재시도·자기개선* 같은 "Agentic"의 핵심 행동 패턴은 아직 약함.

## 2. 개선 포인트 (우선순위 순)

### 2.1 [우선순위 1] 자동 피드백 재작성 루프 (기능3 ↔ 기능2)

**문제**
현재 기능3에서 `severity: critical` 경고가 나와도, 팀장이 수동으로 텍스트를 고쳐야만
"수정 내용으로 품질 재체크" 버튼을 통해 기능3이 재실행됨. 기능2(초안 생성 에이전트)는
자신이 만든 결과에 대한 피드백을 받아 스스로 개선할 기회가 없음.

**제안**
- Orchestrator에 **"품질 검증 루프"** 추가:
  ```
  기능2 초안 생성
    → 기능3 품질 체크 (자동, Human Gate 이전에 1회 선행 실행)
    → critical 경고가 있으면:
         기능2에게 "checkers 결과 + suggestion"을 컨텍스트로 추가 전달 → 재작성
         → 기능3 재실행 (최대 N회, 기본 2회)
    → 통과하거나 N회 도달 시 → Human Gate로 진입 (팀장은 이미 1차 정제된 초안을 검토)
  ```
- 종료 조건: `overall_passed == True` 또는 반복 횟수 한도(`MAX_QUALITY_RETRY`, 기본 2)
- UI: 탭3에 "🔁 자동 재작성 N/2회 진행 중" 같은 진행 표시 추가
- 기존에 만든 "수정 내용으로 품질 재체크" 버튼은 **자동 루프 종료 후에도 팀장이 추가로
  수동 조정**할 수 있는 보조 기능으로 유지

**구현 난이도**: 중 (orchestrator.py의 Step2~3 사이에 루프 삽입, agent2에 `previous_quality` 인자 추가)

**기대 효과**: 팀장이 받아보는 "1차 초안"의 품질이 즉시 올라가고, 이것이 가장 핵심적인
"Agent가 자기 결과를 평가하고 개선한다"는 Agentic 행동 패턴을 충족시킴.

### 2.2 [우선순위 2] 기능2에 RAG 가이드라인 사전 적용

**문제**
기능3(`validate_grade_comment`)이 RAG로 등급 기준을 조회해 "사후"에 문제를 찾아내는 반면,
기능2는 가이드라인을 참고하지 않고 초안을 작성함 → 같은 문제가 매번 반복 발생할 가능성.

**제안**
- `agent2_draft.py`의 TOOLS에 `retrieve_grade_criteria`, `retrieve_guideline` 추가
- system prompt에 "초안 작성 전, 평가 대상자의 추정 등급에 해당하는 등급 기준과
  피드백 작성 가이드를 먼저 조회하고 그 기준에 맞춰 작성하라" 지시 추가

**구현 난이도**: 낮음 (기존 RAG Index MCP 서버/도구를 그대로 재사용, 기능2 TOOLS 목록과
프롬프트만 수정)

**기대 효과**: 기능3의 `grade_comment` 체커 경고 발생률 감소 → 2.1의 재작성 루프 반복 횟수도 감소

---

### 2.3 [우선순위 3] 승인된 피드백의 장기 메모리화

**문제**
`eval_history.json`은 `generate_mock_data.py`가 생성한 정적 데이터. 올해 팀장이 승인한
`draft`가 내년 평가의 "과거 피드백" 입력으로 자동 반영되지 않음.

**제안**
- 파이프라인 완료(`status="done"`) 시점에, 최종 `draft`(팀장 수정 반영본)를
  `data/mock/eval_history.json`에 `{emp_id, year, ...}` 레코드로 append
  (또는 별도 `data/mock/feedback_archive.json` 신설 후 `eval_history_server.py`가
  두 파일을 병합 조회)
- 기능2의 `get_past_feedback` 호출 시 직전 연도 실제 승인 피드백이 조회되도록 함

**구현 난이도**: 낮음~중 (저장 로직 1곳 추가 + `eval_history_server.py` 조회 범위 확장)

**기대 효과**: "작년에 지적된 개선점이 올해 어떻게 반영되었는가"를 기능2가 자연스럽게
언급할 수 있게 되어, 1회성 피드백이 아닌 **연속성 있는 성장 추적**이 가능해짐.

### 2.4 [우선순위 4] 데이터 결손에 대한 적응적 처리

**문제**
CLAUDE.md의 엣지케이스("1on1 이력이 Q1만 있는 직원 → 누락 분기는 '기록 없음' 처리")는
현재 LLM 프롬프트의 일반적인 지시에 의존. 실제로 분기 데이터가 비어 있을 때
기능1이 일관되게 "기록 없음"으로 처리하는지 검증된 테스트 케이스가 없음.

**제안**
- `generate_mock_data.py`에 "Q1만 데이터 존재" 같은 결손 케이스를 가진 테스트 직원 추가
  (또는 기존 직원 1명의 데이터를 일부러 결손시킨 별도 테스트 픽스처)
- 기능1 결과(`summary.json`)에 `data_completeness: {"Q1": true, "Q2": false, ...}` 같은
  메타데이터 필드를 명시적으로 추가하고, 기능2/기능4가 이를 참고해 "데이터 없음" 문구를
  일관되게 노출
- Phase 7(엣지 케이스 테스트)에 자동화된 검증 스크립트 추가

**구현 난이도**: 중 (mock 데이터 + 출력 스키마 확장 + 검증 스크립트)

**기대 효과**: 운영 환경에서 실제로 빈번하게 발생할 "데이터 누락" 상황에 대한 신뢰성 확보

---

### 2.5 [참고] Orchestrator를 LLM 기반으로 전환하는 것은 권장하지 않음

"진짜 Agentic"을 위해 Orchestrator 자체를 LLM이 동적으로 라우팅하게 만드는 방안도
고려할 수 있으나:
- 현재 흐름(1→2→3, 4는 병렬)은 **업무 규칙상 고정**되어 있어 동적 라우팅의 이점이 적음
- LLM 기반 오케스트레이션은 디버깅 난이도·비용·지연시간을 늘리고, 실패 시 흐름 추적이
  어려워짐 (CLAUDE.md의 에러 처리 전략과도 상충)

→ **오케스트레이션은 결정론적으로 유지**하고, "에이전틱함"은 2.1(자기 개선 루프)처럼
**각 에이전트 내부/에이전트 간 피드백 루프**로 확보하는 방향을 권장.
